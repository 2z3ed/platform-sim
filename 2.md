# Conversation Studio

## 1. 文档目标

本文档定义一个“多轮对话仿真工作台（Conversation Studio）”的产品形态、前端布局、API 设计、状态机、后端编排和分阶段实施方案。

该工作台的目标不是单纯生成一条用户问题，而是提供一个可控、可观测、可回放的前端界面，让开发者或测试人员可以：

1. 手动设置用户情绪、问题类型、平台、轮次等参数；
2. 自动或半自动生成真实用户问题；
3. 查看“发送出去的用户内容”；
4. 查看“系统/客服/official-sim 返回的回复内容”；
5. 查看每轮背后的工具调用、状态摘要、API 请求响应和 artifacts；
6. 在中台未完全开发好的情况下，先完成“多轮问题生成 + 上下文更新 + 回复接回”的完整链路。

---

## 2. 一句话定位

Conversation Studio =  
**用户仿真控制台 + 聊天工作台 + 调试观察台**

它不是普通客服坐席台，也不是单纯的 prompt playground。  
它是一个同时服务于：

- 用户问题生成
- 多轮对话推进
- official-sim 联调
- unified 层联调
- 回归测试演示

的前端工作台。

---

## 3. 为什么这个前端必须同时显示“发出去的内容”和“回复内容”

如果界面只能看到“生成了什么用户问题”，你只能验证一句话像不像真实用户。  
但你真正需要验证的是：

1. 用户这句话是不是基于当前订单/物流/退款状态合理生成的；
2. 这句话发出去后，official-sim / reply adapter 那边回了什么；
3. 回完以后，为什么下一轮继续追问或结束；
4. 多轮链路是否符合真实平台和客服系统的行为预期。

因此，Conversation Studio 必须同时具备：

- **用户消息可见**
- **系统/客服/official-sim 回复可见**
- **每轮上下文变化可见**
- **背后的事实与工具调用可见**

## 4. 页面总体结构

建议采用三栏布局：

### 4.1 左侧：场景控制区
作用：配置这一轮或这一段多轮对话“怎么生成”。

建议包含以下控件：

- 平台选择：
  - jd
  - taobao
  - douyin_shop
  - wecom_kf
  - xhs
  - kuaishou
- 用户选择
- 订单选择
- 会话 ID
- 场景选择：
  - 待发货
  - 物流中
  - 已签收未收到
  - 退款处理中
  - 售后待审核
  - 转人工
- 情绪选择：
  - calm
  - impatient
  - angry
- 问题类型：
  - 查订单
  - 催发货
  - 查物流
  - 没收到货
  - 退款/售后
  - 商品咨询
  - 支付/账户
  - 转人工
  - 情绪升级
- 最大轮次：
  - 1 / 3 / 5 / 10
- 模式：
  - 手动单轮
  - 手动多轮
  - 自动多轮
- 生成选项：
  - 是否强制带订单号
  - 是否允许重复追问
  - 是否允许情绪升级
  - 是否允许转人工
  - 是否使用 official-sim 回复
  - 是否使用 stub reply
  - 是否启用错误注入

### 4.2 中间：聊天窗口
作用：像客服对话界面一样，显示整段多轮对话。

建议展示：

- 用户消息
- 系统/客服/official-sim 回复
- 每轮时间戳
- 当前轮次
- 当前对话状态标签（进行中 / 已结束 / 等待人工等）

这个区域的体验应该接近真实客服聊天窗口，但不需要一开始就做成完整坐席台。

### 4.3 右侧：事实 / 调试区
作用：展示这轮生成背后的“为什么”。

建议分成 4 个折叠面板：

#### A. 当前事实摘要
- 订单摘要
- 物流摘要
- 退款摘要
- 最近会话摘要
- 当前状态标签

#### B. Agent 决策摘要
- 当前意图 intent
- 当前情绪 emotion
- 选中的用户 / 订单 / 售后单
- reason / rationale

#### C. 工具调用
- 本轮调用了哪些 tools
- 每个 tool 的参数
- 每个 tool 的返回结果摘要

#### D. API / Artifact 调试
- 调了哪个 reply adapter
- official-sim 请求
- official-sim 返回
- 生成的 artifact ID / run ID / step no
- 错误注入 / 重放情况

## 5. 两种核心使用模式

## 5.1 模式 A：手动控制模式
适合当前阶段：中台还没完全开发好，但你需要先把“多轮问题生成”做出来。

### 典型流程
1. 选平台
2. 选用户
3. 选订单
4. 选情绪
5. 选问题类型
6. 设轮次
7. 点“生成一轮”或“生成多轮”

### 特点
- 用户可控性高
- 便于对比不同情绪、不同类型、不同平台的话术
- 不依赖完整中台即可先跑

## 5.2 模式 B：自动场景推进模式
适合后期联调与回归。

### 典型流程
1. 选平台
2. 选场景
3. 选最大轮次
4. 选是否自动推进状态
5. 点“开始仿真”

### 特点
- Agent 自动决定每轮问什么
- 调用 tools 读取用户视角事实
- 决定下一轮是否继续
- 自动接 official-sim 返回内容
- 更接近真实联调

## 6. 后端职责边界

Conversation Studio 前端不直接调 LLM。  
它应通过统一后端接口调用一个 `conversation-studio-service` 或直接调 `user-simulator-agent` 的 façade。

### 前端只关心三类能力
1. 创建 run
2. 推进一轮
3. 拉取聊天/事实/artifacts

### 后端负责
- 调 user simulator agent
- 调 tools
- 调 official-sim / reply adapter
- 更新上下文
- 记录 artifacts
- 生成 report

## 7. API 设计草图

建议新增一组专门服务前端的 façade 接口：

### 7.1 创建一段会话仿真
`POST /conversation-studio/runs`

请求示例：

```json
{
  "platform": "taobao",
  "mode": "manual",
  "user_id": "tb_user_001",
  "order_id": "TB_ORDER_001",
  "conversation_id": "conv_001",
  "scenario_name": "wait_ship_basic",
  "emotion": "impatient",
  "issue_type": "ask_order_status",
  "max_turns": 5,
  "use_official_sim_reply": true
}

返回示例：

{
  "run_id": "cs_run_001",
  "status": "created",
  "platform": "taobao",
  "conversation_id": "conv_001"
}
7.2 单步推进一轮

POST /conversation-studio/runs/{run_id}/next

请求示例：

{
  "mode": "manual",
  "override": {
    "emotion": "angry",
    "issue_type": "ask_refund_progress"
  }
}

返回示例：

{
  "run_id": "cs_run_001",
  "turn_no": 2,
  "user_message": {
    "text": "我的退款怎么还没到账？",
    "intent": "ask_refund_progress",
    "emotion": "angry"
  },
  "reply_message": {
    "text": "您好，已为您查询，退款正在处理中。",
    "source": "official-sim"
  },
  "continue_suggested": true
}
7.3 获取整段聊天记录

GET /conversation-studio/runs/{run_id}

返回：

run 基本信息
所有消息
当前状态摘要
当前轮次
是否结束
7.4 获取事实与调试信息

GET /conversation-studio/runs/{run_id}/debug

返回：

当前订单摘要
当前物流摘要
当前退款摘要
最近 tools 调用
本轮 artifacts
official-sim request / response 摘要
7.5 获取报告

GET /conversation-studio/runs/{run_id}/report

返回：

轮次摘要
用户问题分布
情绪变化
系统回复来源
open issues
expected vs actual
8. 多轮状态机设计

建议不要把多轮逻辑做成“一条线到底”，而是拆成两段循环：

8.1 用户循环
LOAD_CONTEXT
SELECT_CASE
DECIDE_INTENT
CALL_USER_TOOLS
GENERATE_USER_MESSAGE
EMIT_USER_MESSAGE
8.2 系统反馈循环
CALL_REPLY_ADAPTER
PARSE_REPLY
UPDATE_CONTEXT
DECIDE_CONTINUE
8.3 状态图（简化）
START
  ↓
LOAD_CONTEXT
  ↓
SELECT_CASE
  ↓
DECIDE_INTENT
  ↓
CALL_USER_TOOLS
  ↓
GENERATE_USER_MESSAGE
  ↓
EMIT_USER_MESSAGE
  ↓
CALL_REPLY_ADAPTER
  ↓
PARSE_REPLY
  ↓
UPDATE_CONTEXT
  ↓
DECIDE_CONTINUE
  ├── continue → DECIDE_INTENT
  └── end      → END
9. LangGraph 节点设计

建议节点如下：

nodes = {
    "load_context": 读取 run、用户、订单、最近消息、当前状态,
    "select_case": 选择本轮围绕哪个订单/退款/物流/会话展开,
    "decide_intent": 结构化输出本轮意图、情绪、是否继续,
    "call_user_tools": 调 summary tools 获取用户视角事实,
    "generate_user_message": 渲染自然语言用户消息,
    "emit_user_message": 把消息写入会话,
    "call_reply_adapter": 调 official-sim 或 stub 获取回复,
    "parse_reply": 标准化回复内容,
    "update_context": 更新上下文和轮次,
    "decide_continue": 判断是否继续下一轮
}
10. Reply Adapter 设计

这是当前阶段的关键，因为你说中台还没完全开发好。

10.1 统一抽象

定义一个 reply adapter 接口：

class ReplyAdapter(Protocol):
    def get_reply(self, run_id: str, message: str, context: dict) -> dict:
        ...
10.2 两个实现
A. OfficialSimReplyAdapter

主路径。
调用 official-sim-server 或现有中台 mock / unified 接口，返回真实联调意义上的回复和状态更新结果。

B. StubReplyAdapter

兜底路径。
当中台某段还没 ready 时，用规则/fixture 返回一个临时回复，用来先跑通多轮流程。

10.3 规则
P0 允许 stub
P1 优先 official-sim
最终联调只认 official-sim / unified
11. 工具设计（给 User Simulator Agent）

建议对 LLM 暴露的工具只包含“用户视角事实”工具：

list_user_orders(user_id)
get_order_summary(order_id)
get_shipment_summary(order_id)
get_refund_summary(order_id)
get_recent_messages(user_id, conversation_id)
emit_user_message(conversation_id, message, metadata)

不要把下面这些暴露给 user LLM：

advance_platform_state
inject_error
emit_platform_push
persist_snapshot

因为这些是系统内部动作，不是用户会做的动作。

12. 结构化输出设计

建议两阶段输出。

12.1 决策层输出
{
  "selected_user_id": "tb_user_001",
  "selected_order_id": "TB_ORDER_001",
  "intent": "ask_order_status",
  "emotion": "impatient",
  "should_continue": true,
  "tool_calls_planned": [
    {
      "name": "get_order_summary",
      "arguments": { "order_id": "TB_ORDER_001" }
    }
  ],
  "reason": "订单处于待发货，用户最可能催发货"
}
12.2 文本层输出
{
  "user_message": "我的订单 TB_ORDER_001 怎么还没发货？都等好几天了。",
  "intent": "ask_order_status",
  "emotion": "impatient",
  "style": "normal"
}
