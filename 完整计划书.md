# 模拟官方 API 返回内容与推送行为的多平台仿真 Agent 计划书

## 文档目的

本文档定义 official-sim 项目的完整实施计划，包括项目目标、系统架构、技术选型、平台范围、里程碑拆分、验收标准、工程规范和执行约束。

本项目旨在模拟真实用户行为触发后，各平台官方 API 与推送机制的行为表现，供客服中台进行统一处理、联调测试和回归验证。

---

## 1. 项目目标

### 1.1 核心定位

本项目不是客服回复机器人，而是一个「多平台官方行为仿真层」：

- 模拟真实用户行为发生后，平台内部状态如何变化
- 官方 API 会返回什么内容
- 官方会推什么 webhook / callback
- 把这些内容交给客服中台统一处理

### 1.2 项目名称

- official-sim-server
- 或 user-simulator-agent

### 1.3 LLM 使用约束

LLM 只能用于以下场景：

- 计划生成
- 场景编排
- 报告生成
- 测试说明
- 文档整理

禁止使用 LLM 定义「官方真实 payload 真相」。真相只能来自：

- state machine
- fixtures
- validators
- emitters
- repositories

## 2. 系统架构

### 2.1 五层架构

```
┌─────────────────────────────────────────┐
│           前端 / 测试台                   │
├─────────────────────────────────────────┤
│         AI / 规则层                      │
│    (意图识别、工具调用、建议回复)           │
├─────────────────────────────────────────┤
│        Unified / Domain 层              │
│    (统一订单、物流、售后、会话)             │
├─────────────────────────────────────────┤
│         Provider 层                      │
│    (mock / real 可切换)                  │
├─────────────────────────────────────────┤
│       official-sim 层                    │
│   (模拟官方 API 行为、状态机、推送)         │
└─────────────────────────────────────────┘
```

### 2.2 分层原则

- **official-sim-server**：模拟平台行为，维护场景状态机，产出平台侧「真相」
- **providers/***：平台适配器，负责对接或消费 official-sim / real API
- **domain-service / unified**：标准化领域对象
- **ai-orchestrator**：建议回复、规则判断
- **前端**：只调用 unified 层

### 2.3 数据流

```
用户动作
  → official-sim 推进平台状态
  → official-sim 生成官方查询返回与推送工件
  → provider 消费并标准化
  → unified 层暴露统一对象
  → AI / 前端使用统一对象完成建议回复、展示和联调
```

## 6. API 约定

### 6.1 P0 必须实现的核心接口

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | /official-sim/runs | 创建仿真 |
| GET | /official-sim/runs/{run_id} | 查询仿真状态 |
| POST | /official-sim/runs/{run_id}/advance | 推进状态 |
| GET | /official-sim/runs/{run_id}/artifacts | 获取工件 |
| POST | /official-sim/runs/{run_id}/inject-error | 注入错误 |
| POST | /official-sim/runs/{run_id}/replay-push | 回放推送 |
| GET | /official-sim/runs/{run_id}/report | 获取报告 |

### 6.2 接口要求

1. 所有接口返回统一 envelope 或明确 schema
2. 所有错误都有稳定 code / message / request_id
3. 所有接口都要有 pytest
4. 所有接口都要有 README 中的 curl 示例

## 8. 平台规范

### 8.1 淘宝 (taobao)

**必须实现**：

- 订单状态机：WAIT_BUYER_PAY → WAIT_SELLER_SEND_GOODS → WAIT_BUYER_CONFIRM_GOODS → TRADE_FINISHED
- 售后状态机：REFUND_REQUESTED → APPROVED → REFUNDED
- 工件：trade_detail, order_detail, shipment_snapshot, refund_snapshot
- 错误：token_expired, resource_not_found, duplicate_push

### 8.2 抖店 (douyin_shop)

**必须实现**：

- 订单状态机：CREATED → PAID → WAIT_SHIP → SHIPPED → COMPLETED
- 退款状态机：REQUESTED → REVIEWING → APPROVED → REFUNDED
- 签名验证：timestamp, sign 校验
- 工件：order_detail, push_order_status_changed
- 错误：invalid_signature, timestamp_out_of_window

### 8.3 企微客服 (wecom_kf)

**必须实现**：

- 会话状态机：NEW → QUEUED → ACTIVE → TRANSFERRED → ENDED
- 消息链路：callback → sync_msg → event_message
- 工件：conversation_snapshot, callback_event, sync_msg_page
- 错误：invalid_cursor, msg_code_expired, conversation_closed
