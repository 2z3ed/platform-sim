## 4. 推荐技术栈

## 4.1 前端框架：Next.js
推荐原因：

- 原生支持 Server Components、Suspense、Server Functions
- 原生支持 streaming

- 与 React 生态和 TypeScript 配合成熟

因此，本项目推荐基于 Next.js App Router 开发。:contentReference[oaicite:5]{index=5}

## 4.2 UI 核心：React + TypeScript
React 官方文档明确指出：React 是 UI 组件库，本身不规定路由与数据获取；如果你要构建完整应用，建议使用全栈 React 框架，如 Next.js。React 组件模型本身也非常适合把你的页面拆成聊天区、控制区、调试区等独立组件。:contentReference[oaicite:6]{index=6}

## 4.3 AI 交互层：OpenAI Agents SDK 或 Vercel AI SDK
### 优先思路
- 如果你更重视 **agent/workflow/tool-calling 一致性**，优先考虑 **OpenAI Agents SDK**
- 如果你更重视 **前端聊天 UI、流式体验、Next.js 集成速度**，优先考虑 **Vercel AI SDK**

OpenAI 官方明确提供 Agents SDK，用于构建 agents；Vercel AI SDK 则是一个面向 React、Next.js、Vue、Svelte 和 Node.js 的 TypeScript toolkit。:contentReference[oaicite:8]{index=8}

### 本项目建议
P0 阶段可以：
- **前端 UI 层优先用 Next.js + React + TypeScript**
- **后端 agent/workflow 层仍由 Python + LangGraph 负责**
- 前端先通过 HTTP/SSE/WebSocket 消费后端输出
- Agents SDK / Vercel AI SDK 作为增强层，而不是强制第一天就绑死

这样风险最小。

## 4.4 后端 API 层：FastAPI
FastAPI 官方明确定位为现代、快速（高性能）的 Python API 框架，基于标准类型提示；还自带基于 OpenAPI 的交互式文档界面。这个特性非常适合你当前已有的 official-sim、conversation-studio、user-simulator-agent 这些服务。:contentReference[oaicite:9]{index=9}

## 4.5 编排层：LangGraph
LangGraph 官方将其定位为低层 agent orchestration framework/runtime，面向 long-running、stateful agents；同时它还有 frontend patterns，可以实时可视化 graph 执行和节点状态。你的多轮用户仿真、state machine、tool-calling loop、本质都非常适合放在 LangGraph 后端。:contentReference[oaicite:10]{index=10}

# Frontend Pages

## 1. 文档目标

本文档定义 `conversation-studio-web` 的页面结构、页面职责、组件边界和 P0/P1 页面范围。

目标：
1. 明确前端不是普通网页，而是 AI 工作台。
2. 明确每个页面负责什么，不负责什么。
3. 让 Codex 能按页面维度推进开发，而不是一口气把所有 UI 混写在一个页面里。

---

## 2. 页面总览

建议前端包含以下页面：

1. `/`
2. `/studio`
3. `/runs/[runId]`
4. `/reports/[runId]`
5. `/settings`（可选，P1）

---

## 3. 页面定义

## 3.1 `/`
### 用途
项目首页 / 导航页 / 快速入口页。

### P0 必须包含
- 项目标题
- Conversation Studio 入口
- 最近 run 列表（可选）
- 一个“快速开始”说明
- 当前系统状态简要说明（例如：official-sim 是否可用）

### P0 不要求
- 完整统计面板
- 复杂仪表盘
- 大量 marketing 内容

### 组件建议
- `AppHeader`
- `QuickStartCard`
- `EntryCard`
- `RecentRunsCard`

---

## 3.2 `/studio`
### 用途
主工作台页面，负责：
- 配置仿真参数
- 启动 run
- 推进一轮或多轮
- 展示聊天记录
- 展示 debug / facts / artifacts

这是 P0 最核心的页面。

### 页面布局
三栏布局：

#### 左侧：Control Panel
功能：
- 平台选择
- 用户选择
- 订单选择
- 场景选择
- 情绪选择
- 问题类型选择
- 最大轮次设置
- 手动/自动模式切换
- reply source 切换（official-sim / stub）
- 错误注入开关（P1 可增强）

建议组件：
- `PlatformSelector`
- `ScenarioSelector`
- `EmotionSelector`
- `IssueTypeSelector`
- `ModeSelector`
- `TurnsSelector`
- `ReplySourceToggle`
- `RunActions`

#### 中间：Conversation Panel
功能：
- 展示用户消息
- 展示系统/客服/official-sim 回复
- 展示轮次
- 展示会话状态
- 支持“生成下一轮”“自动跑完”“停止”

建议组件：
- `ConversationTimeline`
- `ChatBubble`
- `TurnHeader`
- `ConversationToolbar`

#### 右侧：Debug / Facts Panel
功能：
- 展示当前订单/物流/退款摘要
- 展示 structured decision
- 展示 tool calls
- 展示 reply adapter 调用结果
- 展示 artifacts / run / step 摘要

建议组件：
- `FactPanel`
- `DecisionPanel`
- `ToolCallsPanel`
- `ReplyTracePanel`
- `ArtifactPanel`

### P0 必须具备
- 创建 run
- 展示至少一轮用户消息和回复
- 能推进下一轮
- 能看到当前事实摘要
- 能看到当前 decision / tool calls
- 能看到 reply source

### P0 不要求
- 图谱可视化
- 多窗口并行 run
- 完整 report 图表
- 自定义 persona 编辑器

---

### P0 必须具备

### P1 可增强

# Frontend API Contract

## 1. 文档目标

本文档定义 `conversation-studio-web` 前端会调用的后端 API 契约。

目标：
1. 固定前端与后端的交互边界；
2. 避免前端直接拼凑平台真相；
3. 让 Codex 在做前端和后端 façade 时有明确接口；
4. 让页面与 run / turn / debug / report 的数据结构可预期。

---

## 2. 设计原则

### 2.1 前端只调 façade API
前端不直接调用底层平台 mock/provider 细粒度接口。  
优先调用：

- `conversation-studio-service`
- `official-sim-server` 暴露的 façade
- `user-simulator-agent` façade

### 2.2 一次对话 run 是核心对象
前端围绕 `run` 工作，而不是围绕一堆零散接口工作。

### 2.3 结构化优先
所有关键响应都应有稳定 schema，前端不要猜字段。

### 2.4 前端只读“用户视角事实”
前端展示的 facts 是整理后的摘要，不是底层 raw payload 大 dump。

---
