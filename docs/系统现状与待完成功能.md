# Platform-Sim 系统现状与待完成文档

## 一、项目目标

**核心定位**: 纯模拟系统，不处理任何真实数据，专门模拟"官方平台API返回"，供中台系统开发测试使用。

**最终效果**:
```
┌─────────────────────────────────────────────────────────────┐
│  LLM (扮演用户)           official-sim-server (扮演官方)    │
│       │                           │                         │
│  "我要退款"              返回京东官方退款API格式            │
│  "查物流"                返回官方物流查询格式               │
└─────────────────────────────────────────────────────────────┘
```

---

## 二、参考设计文档

**重要参考**: `1.md` 中的 User Simulator Agent 设计

核心要点:
1. **三层架构**: World Engine / User Agent / Platform Layer
2. **两阶段输出**: 结构化意图 + 用户话术
3. **用户视角工具**: 只暴露用户能感知的信息
4. **Evaluator**: 验证消息是否符合真实用户

## 四、三层职责边界

| 层级 | 谁负责 | 做什么 | 不做什么 |
|-----|-------|-------|---------|
| **World Engine** | 状态机/fixtures | 真相(订单状态/物流/退款) | 不生成用户消息 |
| **User Agent** | LLM | 生成用户消息 | 不决定世界真相 |
| **Platform Layer** | fixtures | 官方API格式 | 不生成对话 |

## 七、Prompt 约束

### 7.1 核心约束

```
1. 只能基于工具结果说话
   "你不能编造订单状态、物流状态、退款状态。
    你只能依据工具返回结果生成用户消息。"

2. 只模拟用户，不模拟客服
   "你扮演用户，不要替客服解释，不要输出客服建议，
    不要调用系统内部管理动作。"

3. 先判断用户最关心的问题，再决定是否调用工具
   "先判断用户最关心的问题，再决定是否调用工具"

4. 输出先结构化，再生成文本
   "先产出结构化意图，再生成用户话术"
```

### 7.2 Prompt 模板

```
你是用户行为模拟器。你的任务是根据用户信息生成真实的用户消息。

## 用户信息
- 用户ID: {user_id}
- 用户名: {user_name}
- 平台: {platform}

## 可用工具
{tools_schema}

## 核心约束
1. 你不能编造订单状态、物流状态、退款状态

1. 选择一个用户和该用户的订单
```

## 八、Evaluator (消息校验)

### 8.1 校验规则

| 校验项 | 说明 |
|-------|------|
| 引用不存在的order_id | 错误 |
| 说与refund_status不一致的话 | 错误 |
| 用了平台看不到的内部字段 | 错误 |
| 长度超标 | 错误 |
| 没有情绪/意图标签 | 错误 |

### 8.2 校验实现

```python
def validate_message(message: dict, tool_results: list) -> ValidationResult:
    errors = []

    # 检查order_id是否存在
    if message.get("selected_order_id"):
        if not order_exists(message["selected_order_id"]):
            errors.append("引用了不存在的order_id")

    # 检查与工具结果是否矛盾
    if message.get("intent") == "ask_refund":
        if not has_refund_record(message["selected_order_id"]):
            errors.append("用户说退款但工具显示无退款记录")

    # 检查是否用了内部字段
    internal_fields = ["advance_order_state", "emit_platform_push"]
    if any(f in str(message) for f in internal_fields):
        errors.append("使用了系统内部字段")

    return ValidationResult(passed=len(errors) == 0, errors=errors)
```

## 十三、常见错误 (避免)

| 错误 | 后果 | 正确做法 |
|-----|------|---------|
| 让Agent决定"世界真相" | 不再是官方仿真 | 真相来自World Engine |
| 直接暴露平台底层原始字段 | 生成的话术不像人话 | 先做summary tool |
| 一步输出最终文本 | 不利于测试审计 | 两阶段输出 |
| 工具不给LLM暴露 | Agent无法获取数据 | 分用户工具/系统工具 |

### 多轮对话状态机设计（待实现）

#### 状态定义
```
States:
- IDLE: 初始化
- SELECT_USER: 选择用户和订单
- THINK: LLM决策下一步做什么
- CALL_TOOL: 调用工具查询数据
- GENERATE_MSG: 生成用户消息
- CALL_API: 调用official-sim-server获取客服回复
- UPDATE_CONTEXT: 更新对话上下文
- DECIDE_CONTINUE: 判断是否继续
- END: 结束
```

#### 状态流转
```
START → IDLE → SELECT_USER → THINK → CALL_TOOL → GENERATE_MSG
                                              ↓
                                       CALL_API → UPDATE_CONTEXT
                                              ↓
                                       DECIDE_CONTINUE
                                         ↓           ↓
                                      [继续]      [结束]
                                         ↓           ↓
                                       THINK        END
```

#### 节点设计
```python
nodes = {
    "select_user": 选择用户和订单,
    "think": LLM决策下一步(查询/生成消息/结束),
    "call_tool": 调用FixtureLoader工具,
    "generate_message": 生成用户消息,
    "call_official_api": 调用official-sim-server,
    "update_context": 更新对话历史和状态,
    "decide_continue": 判断是否继续对话,
}
```

#### 待讨论问题
1. 客服回复来源: official-sim-server API vs LLM生成
2. 订单状态更新机制
3. 最大轮次限制 (建议5-10轮)

## 12.3 当前问题

### ❌ 核心问题: LLM API 无法访问

**现象**:
```
SSLError: HTTPSConnectionPool(host='dashscope.aliyuncs.com', port=443)
```

**原因分析**:
- 当前环境网络受限，无法访问外网
- `ping dashscope.aliyuncs.com` 返回 `198.18.0.56` (异常 IP 段)
- `curl https://www.baidu.com` 返回网络不通

**影响**:
- 用户消息生成依赖 LLM，当前只能使用 Fallback 模式
- Fallback 模式使用预设模板，消息较为单调

**解决方案**:
1. 检查/配置网络代理

3. 部署到可访问外网的环境
