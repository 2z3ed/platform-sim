你要找的 mock 代码，主要分成两层：

Mock 服务本体

JD：/mock/jd/oauth/token、/mock/jd/orders/{order_id}、/mock/jd/shipments/{order_id}、/mock/jd/after-sales/{after_sale_id}。
抖音店铺：/mock/douyin-shop/auth/token、/mock/douyin-shop/orders/{order_id}、/mock/douyin-shop/refunds/{after_sale_id}、/mock/douyin-shop/products/{product_id}。
企微客服：/mock/wecom-kf/token、/mock/wecom-kf/messages/sync、/mock/wecom-kf/service-state/trans、/mock/wecom-kf/event-message/send。

这些接口背后的数据源也很清楚：都在 apps/mock-platform-server/app/data/ 下面，按 jd、douyin_shop、wecom_kf 分目录存 JSON fixture。比如 JD 的 order_sample.json、shipment_sample.json、after_sale_sample.json，抖音的 order_sample.json、refund_sample.json，企微的 message_sample.json、service_state_sample.json 等。

结论：可以开发 agent，而且很适合。
但要区分两件事：

做一个“基于 mock 的联调/回归/演练 agent”——现在就可以做。因为 provider mock 层已经把平台差异封装了一层，你可以直接围绕这层做场景驱动、自动验证、异常注入。
做一个“接近真实订单接入行为的仿真 agent”——也可以做，但当前 mock 还不够，需要扩展。因为现在大多数接口本质上是“读取静态 JSON，再把传入的 id 覆盖回去”。例如 JD/抖音订单查询基本就是加载 sample 后改 orderId；企微消息接口也只是把部分请求字段回填到响应里，并没有跨请求状态演进。

也就是说，它现在更像“静态 mock + 轻量 echo”，还不是“状态化仿真平台”。从当前代码看，我没有看到订单生命周期推进、webhook 回调、签名校验、幂等、重试、延迟/超时、分页、限流之类的真实接入行为实现；看到的是固定 token 返回、固定样例数据、以及少量字段覆盖。这个判断是基于现有路由和 fixture 的实现方式做出的。

我建议你这样做：

最佳切入层：先不要直接绑死在 mock-platform-server 路由上，而是优先围绕 providers/*/mock/provider.py 或统一 API 层来做 agent。
原因很简单：当前项目已经把 mock provider 和 future real provider 设计成同一模式，等以后真实平台接入落地，你的 agent 可以少改甚至不改，只替换 provider 即可。仓库 README 里也明确暴露了统一 API，如 /api/orders/{platform}/{orderId}、/api/shipments/{platform}/{orderId}、/api/ai/suggest-reply 等，适合做端到端演练。

你这个 agent，最有价值的方向是做成 “场景仿真 agent / 联调 agent”，让它能驱动这些典型链路：

正常链路：已支付 → 已发货 → 运输中 → 签收。
售后链路：仅退款、退货退款、售后驳回、重复申请。
客服链路：企微用户进线 → 查询订单 → 查询物流 → 建议回复。
异常链路：订单不存在、物流超时、平台 token 失效、接口 429、接口超时、部分字段缺失、状态回跳。

而要支持这些，你最好先把 mock 服务补成下面这几个能力：

场景状态机：一个订单不是一份固定 JSON，而是一个 scenario_id + current_state。
场景管理接口：例如创建场景、推进状态、重置场景、查询当前快照。
动态 fixture/模板：按平台、品类、售后类型、风险标签生成不同响应。
事件机制：模拟 webhook / callback，而不是只有轮询 GET。
故障注入：可配置延迟、超时、随机 5xx、签名错误、重复回调。
持久化：把 scenario 存 Redis/Postgres，而不是仅靠内存或静态文件。

但你这里有一个非常关键的边界

不要把这些数据集的 assistant 回答当成系统标准答案。

因为你的系统里，“答案”必须由 API 事实决定。
举例：

用户问“我的订单到哪了？”
问法可以来自 HF 数据集
但答案必须来自 /mock/unified/shipments/{platform}/{orderId} 或平台物流接口。
用户问“退款什么时候到账？”
问法可以来自 HF 数据集
但状态必须来自 /mock/unified/after-sales/{platform}/{afterSaleId}。
用户催发货、要求人工、情绪升级

所以最合理的用法是这套流程
A. 先由 API 决定“这个用户现在有资格问什么”

例如：

订单 WAIT_SELLER_STOCK_OUT
可以问：怎么还不发货、能不能催单、今天能发吗

也就是说，intent 是 API 状态约束出来的，不是数据集随便抽。

B. 再由 HF 数据集提供“表达方式”

例如同一个 intent ask_shipment_delay，可以从数据集里抽不同表述：

正常型：“我想查一下物流到哪了”
催促型：“都好多天了还没到，到底什么情况”
激动型：“系统写已送达但我根本没收到，你们查一下”

这一步最好按：
intent + emotion + urgency + turn_index + persona

因为你不是做通用电商客服，而是“多平台用户仿真”。
所以还要再加一层平台风格：

京东用户更像“催发货 / 查物流 / 售后”
抖店用户更容易出现“直播间下单 / 地址改不了 / 催退款”

用户行为

平台内部状态
→ 订单状态、物流节点、售后状态、会话状态发生变化

官方平台输出
→ 订单查询接口、物流接口、售后接口、消息同步接口、事件推送 / webhook

你的中台处理
→ 统一字段映射、上下文聚合、规则判断、AI 建议回复、人工发送

从官方文档看，淘宝、抖店、企业微信客服、小红书都明显是 “状态机 + 查询 API + 主动推送/事件回调” 的组合；京东公开可检索文档也明确开放了商品、交易、物流、售后等能力，并且物流与发货能力是独立能力域。

先说核实后的几个关键事实。

淘宝 / 阿里开放平台
淘宝订单模型官方就是 trade（主订单）/ order（子订单） 两层；公开文档给了典型正向状态链路：WAIT_BUYER_PAY -> WAIT_SELLER_SEND_GOODS -> WAIT_BUYER_CONFIRM_GOODS -> TRADE_FINISHED。官方还明确建议优先用 RDS 订单推送 来拿订单实时变化，而不是只靠轮询；批量查询接口返回的是部分数据，完整详情要再调 taobao.trade.fullinfo.get。售后侧也不是一个静态状态，而是单独的退款/退货流程，且不同状态下商家有响应时效，不处理会触发自动同意等业务后果。

抖店开放平台
抖店订单详情查询是正式 API，公共参数里就包含 method / app_key / access_token / timestamp / v / sign / sign_method，并要求对参数签名；官方调用指南写明签名算法支持 hmac-sha256（推荐）和 md5。同时，抖店还有独立的 消息推送服务，官方明确说订单、退款、商品等事件可以主动推送到开发者配置地址，开发者不必只靠轮询；平台会先发测试消息验证地址，请求头里还带 event-sign 等字段做校验。

企业微信客服
企微客服不是“平台直接把完整聊天记录推给你”，而是 先回调事件，再让你用 kf/sync_msg 主动拉取消息。官方镜像文档显示：当客户或接待人员发消息、有行为动作时，企业微信会把事件回调到企业 URL；随后企业再用 sync_msg 去读取最近 3 天内的具体消息和事件。消息和事件类型非常丰富，包括文本、图片、语音、菜单、小程序，以及 enter_session、session_status_change、msg_send_fail 等事件。会话流转靠 kf/service_state/trans，事件响应消息靠 kf/send_msg_on_event，而且不同事件 code 有时效和条数限制。

小红书开放平台
小红书官方文档写得很直接：订单获取模式是 轮询订单列表，单个商家建议轮询频率 不小于 5 分钟；订单到待发货后，再拉订单详情、收件人信息、跨境申报信息等。售后侧也有独立接口和消息体系，官方公告专门说明过售后接口与售后消息的更新，列出了如 msg_after_sale_update、msg_after_sale_closed、msg_after_sale_refund_finished 等消息事件。

京东开放平台
京东公开首页和能力页能确认，它对外开放的是商品、交易、仓储、物流、售后等能力；物流轨迹查询和订单发货也是独立能力。公开可检索页面里，物流轨迹能力明确提到可按订单号/运单号查配送节点、配送时间等，订单发货能力明确是独立履约能力。
但要实话实说：京东很多字段级 API 明细页公开搜索可见性不如淘宝/抖店高。我能从公开页面确认能力域和部分能力说明，但更细的字段级接口内容，通常需要进控制台或更深层文档页查看。

这套核实结果，和你仓库里现在的 OpenAPI 契约是方向一致、粒度不足的关系。你现在这份契约已经把平台 mock、统一订单/物流/售后、企微客服、webhook 回放这些接口面定义出来了，也明确写了“先 mock provider，未来替换 real provider”。但它本质还是一层中间契约，不是对各平台官方协议的严格镜像。

所以，接下来这个 agent 要怎么开发，我建议你按 “官方行为仿真” 而不是 “静态数据 mock” 去做。

一、先把仿真对象拆成 4 层
1. 业务状态层

这是最核心的。真实用户行为会改变：

订单状态
物流状态
售后状态
会话状态
客服接待状态

没有这个状态机，你只能返回固定 JSON，不叫“模拟官方 API 行为”。

2. 官方输出层

同一个状态变化，会通过不同平台的不同载体暴露出来：

查询接口返回
订单列表增量
详情接口字段变化
退款接口字段变化
webhook / 推送消息
企微事件回调 + sync_msg

也就是说，agent 的输出应该是“平台会吐出什么接口返回 / 什么推送消息”，不是一句自然语言。

3. 协议行为层

官方接口不只有业务字段，还有协议特征：

OAuth / access token
请求签名
时间戳
幂等或 msg_code/code 约束
推送签名
地址校验
轮询频率建议
48 小时会话窗口 / 消息条数限制

这层如果不模拟，开发联调时会丢失很多真实接入问题。淘宝 OAuth2.0、抖店签名、企微事件 code 与会话窗口限制，官方都写得很清楚。

4. 统一中台映射层

最后才是你的中台去把这些平台返回标准化成统一模型。你上传的契约已经定义了 OrderCore / ShipmentResponse / AfterSaleResponse / SuggestReplyResponse 这类统一层对象，所以这个层不用推翻，只要在上游补足“官方行为仿真”即可。

二、agent 不该直接“写死接口”，而该驱动场景状态机

最合理的核心结构是：

用户动作 -> 平台状态机推进 -> 生成官方查询返回 + 生成官方推送事件 -> 你的中台消费

例如一个“催发货”场景：

用户先下单未付款
用户付款
平台订单状态从待付款变成待发货
淘宝/抖店订单详情接口字段变化
平台可能推一条订单状态更新消息
用户发消息“怎么还不发货”
你的中台查统一订单、查物流、生成建议回复

这里真正被 agent 模拟的是第 2 到第 5 步。第 6 步只是附带的人类问法。

三、你的 agent 最好拆成三个子模块
A. Scenario Engine

负责维护场景状态。

建议最少支持这些场景：

已下单未付款
已付款待发货
已发货运输中
已签收但用户称未收到
未发货仅退款
已发货退货退款
售后审核中
售后关闭
转人工
非工作时间进入会话
消息发送失败 / 用户拒收

这些场景都能在官方文档里找到合理依据：淘宝有正向订单链路和退款链路，抖店有订单和退款消息推送，企微有会话状态与消息失败事件，小红书有订单与售后消息。

B. Platform Behavior Profiles

每个平台一份 profile，不只是字段映射，还要包含行为规则：

查询型 or 推送型主导
订单主/子结构
售后是否独立对象
会话是否走回调+拉取
签名算法
推送头校验
token 过期行为
频控/窗口/条数限制
错误码模板

比如：

淘宝 profile：trade/order、RDS 推送优先、详情接口补全
抖店 profile：签名严格、订单/退款/商品消息主动推送
企微 profile：事件回调后 sync_msg 拉取、service_state/trans、send_msg_on_event
小红书 profile：订单列表轮询为主、售后消息独立
京东 profile：订单/物流/售后能力分域，履约与物流能力单独模拟
C. Emitter / Transport Layer

负责把状态机结果“吐”成不同平台的表现形式：

REST 查询响应
webhook POST
callback event
callback 后再供 sync_msg 读取
header/sign 校验失败
token 失效
空结果 / 资源不存在 / 参数错误

这层才是“模拟官方 API 接口行为”的直接实现。

二、这层和你现有项目怎么衔接

这层不要直接服务前端。它的职责是：

维护平台业务状态机
生成“官方查询返回”
生成“官方推送/回调事件”
把这些结果喂给你现有的 /mock/...、/mock/unified/... 或 provider 层

五、核心类怎么设计

这一层最重要的不是路由，而是下面这五种类。

1. PlatformProfile

描述一个平台“像什么”。

from abc import ABC, abstractmethod
from app.models.run import SimulationRun
from app.models.artifacts import RunArtifacts

class PlatformProfile(ABC):
    platform_name: str

    @abstractmethod
    def build_initial_state(self, scenario_name: str) -> dict:
        ...

    @abstractmethod
    def advance_state(self, run: SimulationRun, action: dict | None = None) -> dict:
        ...

    @abstractmethod
    def build_artifacts(self, run: SimulationRun) -> RunArtifacts:
        ...

职责：

定义平台状态机
定义官方字段形态
定义协议差异
定义推送/轮询/回调模式
2. StateMachine

只关心状态推进，不关心 HTTP。

from abc import ABC, abstractmethod

class BaseStateMachine(ABC):
    @abstractmethod
    def transition(self, state: dict, event: str, payload: dict | None = None) -> dict:
        ...

各平台自己实现，比如：

TaobaoStateMachine
JDStateMachine
DouyinStateMachine
WeComKFStateMachine
XHSStateMachine
KuaishouStateMachine
3. ArtifactBuilder

把状态翻译成“官方会返回什么”。

from abc import ABC, abstractmethod
from app.models.run import SimulationRun
from app.models.artifacts import RunArtifacts

class BaseArtifactBuilder(ABC):
    @abstractmethod
    def build(self, run: SimulationRun) -> RunArtifacts:
        ...

举例：

淘宝 builder 会产出 trade/order/refund/push
抖店 builder 会产出 order/refund/product/push + sign headers
企微 builder 会产出 callback + sync_msg buffer + send_msg_on_event prerequisites
4. PushEmitter

负责生成主动推送或回调。

from abc import ABC, abstractmethod
from app.models.run import SimulationRun
from app.models.artifacts import PushArtifact

class BasePushEmitter(ABC):
    @abstractmethod
    def emit(self, run: SimulationRun) -> list[PushArtifact]:
        ...

平台差异：

淘宝：订单推送
抖店：消息推送 + 签名
企微：事件回调
小红书：P0 可先弱化为轮询，不一定强推送
京东 / 快手：P0 可先做占位 emitter
5. ErrorInjector

负责让它更像真实平台，而不是“永远 happy path”。

from abc import ABC, abstractmethod

class BaseErrorInjector(ABC):
    @abstractmethod
    def maybe_inject(self, artifact: dict, state: dict) -> dict:
        ...

统一支持的错误：

token 过期
sign 错误
timestamp 超窗
未授权
资源不存在
空结果
推送重复
推送乱序
推送延迟
频控

这些错误并不是拍脑袋加的：你前面要的六个平台里，淘宝/抖店/企微/小红书都有明显的授权、推送、轮询、状态流特征；抖店还明确有签名和推送校验；企微有 callback -> sync_msg -> send_msg_on_event 这样的链路约束。(developer.alibaba.com

七、最关键的服务类
1. SimulationRunService
class SimulationRunService:
    def __init__(self, run_repo, profile_registry):
        self.run_repo = run_repo
        self.profile_registry = profile_registry

    def create_run(self, platform: str, scenario_name: str, strict_mode: bool) -> SimulationRun:
        profile = self.profile_registry.get(platform)
        state = profile.build_initial_state(scenario_name)
        run = SimulationRun(
            run_id="...",
            platform=platform,
            scenario_name=scenario_name,
            strict_mode=strict_mode,
            state=state,
        )
        return self.run_repo.save(run)

    def advance(self, run_id: str, action: dict | None = None) -> SimulationRun:
        run = self.run_repo.get(run_id)
        profile = self.profile_registry.get(run.platform)
        run.state = profile.advance_state(run, action)
        run.current_step += 1
        return self.run_repo.save(run)

    def artifacts(self, run_id: str) -> RunArtifacts:
        run = self.run_repo.get(run_id)
        profile = self.profile_registry.get(run.platform)
        return profile.build_artifacts(run)

class ProfileRegistry:

LangGraph 官方写得很明确：它是一个 low-level orchestration framework and runtime，适合构建 long-running, stateful agents，重点是 durable execution、human-in-the-loop、stateful workflow。这和你的系统需求高度一致，因为你真正要编排的是“用户动作注入 → 平台状态推进 → 生成官方返回/推送 → 收集工件”，而不是自由探索。

2. 为什么 LangChain 只做薄层

LangChain 官方把自己定位成更上层的框架，适合快速接模型、工具、基础 agent loop；LangGraph 可以单独使用，不强依赖 LangChain。对你来说，LangChain 最合适的角色是：模型接入、工具 schema、prompt 管理、结构化输出封装。不要让 LangChain 直接控制整套仿真状态。

3. 为什么现在不选 DeepAgents

Deep Agents 官方明确说它是建立在 LangChain 核心之上、使用 LangGraph runtime 的 standalone library，特点是规划、文件系统上下文管理、子代理和长期记忆，适合复杂、多步骤、较强自主性的任务。你现在并不需要这些能力；你需要的是少自治、多确定性。

4. 为什么 DeerFlow 2.0 不做主框架

DeerFlow 2.0 官方定位就是 long-horizon super agent harness，强调 sub-agents、memory、sandboxes、skills。这个非常适合以后做大规模测试编排或研究型回归，但不适合当“官方 API 行为仿真内核”。你现在最怕的是不稳定，而 DeerFlow 更适合放到最外层，等你的仿真层跑稳以后再接。

# Official Simulation Server Rules

## Project purpose
This repository implements a stable official-API behavior simulation layer for a multi-platform customer service middleware.

The scope includes:
- JD
- Taobao
- Douyin Shop
- WeCom KF
- Xiaohongshu
- Kuaishou

This layer simulates:
- order state changes
- shipment state changes
- after-sale/refund state changes
- conversation and callback events
- auth/signature/rate-limit/permission failures
- official query responses
- official push/callback artifacts

This layer does NOT:
- replace the existing mock-platform-server
- let the LLM invent official fields
- let the LLM define business truth
- directly optimize for fancy autonomous behavior

## Architecture rules
1. Keep deterministic truth in code.
2. LLM may generate user actions, plans, and reports, but must never generate source-of-truth API payloads.
3. All official payloads must come from:
   - state machine
   - fixtures
   - validators
   - emitters
4. Keep unified-layer compatibility in mind, but do not couple official simulation logic to unified response objects.
5. Prefer explicit typed models over dict-heavy code.
6. One platform profile per platform.
7. Every platform profile must have:
   - profile.py
   - state_machine.py
   - artifact_builder.py
   - error_injector.py
   - fixtures/
8. No hidden global state.
9. Every run must be reproducible from persisted state and fixtures.
10. Every change must come with tests.

## Coding rules
- Python 3.11+
- FastAPI for HTTP layer
- Pydantic models for all request/response/state objects
- Avoid metaprogramming
- Avoid magic strings when enums can be used
- Prefer composition over inheritance, except for clean base interfaces
- Do not introduce background workers in v0.1
- Do not introduce message brokers in v0.1
- Keep the first version synchronous or simple async only

## Stability rules
- Do not add DeerFlow, DeepAgents, or complex memory systems in v0.1
- Do not add RAG in v0.1
- Do not add browser automation in v0.1
- Do not add external search into runtime paths
- Keep the runtime deterministic and debuggable

## API design rules
Implement these route groups first:
- /official-sim/runs
- /official-sim/taobao/*
- /official-sim/douyin-shop/*
- /official-sim/wecom-kf/*
Then:
- /official-sim/xhs/*
- /official-sim/jd/*
- /official-sim/kuaishou/*

## Output quality rules
When making changes:
1. first create files and skeletons
2. then wire routers
3. then implement typed models
4. then implement state transitions
5. then implement artifact generation
6. then add tests
7. then run lint and tests

## Forbidden behaviors
- never replace fixtures with free-form model output
- never hardcode random payload fields without schema
- never skip tests
- never silently change route contracts
- never add dependencies without clear justification

1. 当前项目的目的，先重新定一遍

这个项目不是“把各平台官网 API 原样搬过来”，而是做一个多平台客服中台的模拟契约层 / 数据中台：先用 mock/provider 把系统跑起来，等以后拿到正式平台权限，再把 mock provider 替换成 real provider；上层业务、AI 和前端尽量只依赖统一接口，不依赖某个平台的真实细节。你上传的说明文档和 OpenAPI 都是这个定位：文档明确说“它不是‘真实平台接口集合’，而是你自己的中台模拟契约层”，而 YAML 里也把六个平台、统一层订单/物流/售后、建议回复和 webhook 回放都定义出来了。

这个目标还包含两个非常关键的工程原则：
第一，统一层是产品核心，不是平台层；文档建议先做 /mock/unified/orders、/mock/unified/shipments、/mock/unified/after-sales、/mock/unified/conversations/.../suggest-reply，并明确“前端永远只调 unified 层，不直接调平台层”。第二，provider 要从一开始就 mock/real 分离，未来只替换 provider，不重写上层逻辑。

所以，当前项目的本质可以一句话概括：

做一个能承接六大平台官方行为、并向上提供统一订单/物流/售后/会话能力的客服中台；当前先用 mock 和官方行为仿真跑通，未来再逐个平台切 real provider。

2. 最终系统边界，我建议这样定

系统分成五层最稳。

A. official-sim 层

这层负责模拟官方 API 的行为，不是你的产品接口。
它维护订单状态、物流状态、售后状态、会话状态、鉴权/签名/错误码、推送/回调。

B. provider 层

把 official-sim 或 real platform 返回转换成你内部统一对象。
这里继续保持 provider_mock.py / provider_real.py 的分离。

C. unified/domain 层

暴露统一订单、物流、售后、会话、建议回复接口。
你现有 YAML 已经定义了 /mock/unified/orders/{platform}/{orderId}、/mock/unified/shipments/{platform}/{orderId}、/mock/unified/after-sales/{platform}/{afterSaleId}、/mock/unified/conversations/{conversationId}/suggest-reply，这层不要推翻。

D. AI / orchestration 层

这里只做：

用户动作生成
场景编排
工具调用
报告汇总

不允许它定义“官方返回真相”。

E. 坐席 / 测试前端层

展示：

会话列表
订单侧栏
售后侧栏
AI 建议回复
场景运行状态
webhook 回放与工件

这个五层结构，本质上是在你现有“四层架构”的基础上，加了一个更上游的 official-sim 层。原因是你现在已经明确要做“稳定模拟官方 API 提供的内容”，那就必须把“平台行为仿真”和“中台统一接口”拆开。

3. 技术选型，直接拍板

稳定优先的话，我建议就锁成下面这套。

核心服务框架

FastAPI + Pydantic

原因很简单：FastAPI 的 APIRouter 官方就是拿来把路径操作分组、把应用拆成多文件结构的；你这种按平台拆 jd.py / taobao.py / douyin_shop.py / wecom_kf.py / xhs.py / kuaishou.py / unified.py 的方式，非常适合它。

编排框架

LangGraph

因为 LangGraph 官方定位就是一个低层 orchestration framework and runtime，专门用于构建长时间运行、有状态的 agent/workflow，而且它强调对状态和流程的细粒度控制。你要的不是高自主，而是“用户动作注入 → 状态推进 → 生成官方工件 → 收集结果”的稳定编排，这正是 LangGraph 的强项。

LangChain 的角色

只做薄适配层，不做主控

用它来做：

LLM 接入
tool schema
prompt/template
结构化输出封装

不要把整个仿真真相放给 LangChain/agent loop 决定。

DeerFlow 2.0 / DeepAgents

现在都不进核心运行时

这两个更适合以后做大规模回归、复杂测试探索或研究型 super-agent 外壳。你当前目标是“稳”，不是“聪明”。

LLM

默认：百炼 Qwen3.5-Plus
复杂复盘：百炼 Qwen3-Max

阿里云百炼官方模型页把 Qwen3.5-Plus 放在效果、速度、成本较均衡的位置，并明确写了它纯文本表现媲美 Qwen3-Max、速度更快、成本更低；这很适合你这种需要稳定批量运行、结构化输出和工具调用的系统。

百炼接入方式

OpenAI Chat Completions 兼容接口

百炼官方文档已经提供 OpenAI 兼容的 Chat Completions 入口，这对 LangChain/LangGraph 接入最稳。Function Calling 和 JSON Mode 也都是官方支持的。

七、系统总体架构
7.1 架构原则

整个系统分五层：

official-sim 层：模拟官方 API 行为
provider 层：平台适配，mock/real 可切换
unified/domain 层：统一订单/物流/售后/会话/建议回复
AI/规则层：意图识别、工具调用、建议回复、风险分级
前端/测试台：坐席 UI、场景运行台、回放台

这个分法和你现有说明文档里的四层结构完全兼容，只是在更上游新增一层 official-sim。你文档已经明确说平台适配层、统一领域模型层、AI 与规则层、前端坐席层要分开，并且 unified 层才是产品核心。

7.2 为什么要新增 official-sim

因为你现有 OpenAPI 是“中台对外契约”，而不是“官方行为内核”。OpenAPI 里 unified 层已经定义了标准化订单、物流、售后和建议回复；说明文档也建议前端只调 unified 层。换句话说，当前 /mock/... 和 /mock/unified/... 更适合作为产品层和联调层，而不是直接承担全部官方行为仿真职责。

7.3 数据流

完整数据流应为：

用户动作
→ official-sim 推进平台状态
→ official-sim 生成官方查询返回与推送工件
→ provider 消费并标准化
→ unified 层暴露统一对象
→ AI/前端使用统一对象完成建议回复、展示和联调。

7.4 关键边界

必须强约束：

LLM 不能定义官方 payload 真相
状态机 + fixtures + validators + emitters 才能定义官方返回
AI 只负责动作、计划、报告和解释
provider 是替换点，不是产品核心
unified 层对上保持稳定
八、技术选型
8.1 核心服务：FastAPI + Pydantic

FastAPI 官方明确说明 APIRouter 用于对路径操作分组，方便把应用组织到多个文件中；官方教程也给出“大型应用、多文件”的标准拆分方式。Pydantic 官方说明模型是继承 BaseModel 并用类型注解定义字段的类，非常适合本项目这种强 schema、强类型、强校验的仿真服务。

因此，核心服务采用：

FastAPI 承载 HTTP API
Pydantic 承载状态对象、请求对象、工件对象、数据库 DTO
8.2 编排：LangGraph

LangGraph 官方定位是“低层 orchestration framework and runtime”，用于构建长时、有状态 agent；Graph API 由 State / Nodes / Edges 组成，天然适合“状态推进 + 节点编排”的工作流。对于本项目，它非常适合做：

create run
inject user action
advance state
collect artifacts
evaluate result
summarize report

而不适合的，是把业务真相也交给 agent 自由决定。

8.3 模型接入：LangChain（薄层）

LangChain 官方产品页明确把 LangGraph 定位为可低层控制 agent 的运行时，而 LangChain 用来定制 building blocks。对本项目来说，LangChain 只应该承担：

模型适配
tool schema
prompt/template
structured output 封装

不应该承担官方行为真相控制。

8.4 LLM：百炼 Qwen3.5-Plus 为主，Qwen3-Max 为辅

百炼官方模型页说明：Qwen3.5-Plus 纯文本表现媲美 Qwen3-Max，速度更快、成本更低；Qwen3-Max 更适合复杂任务。对本项目来说，默认主模型应选 Qwen3.5-Plus，只在复杂报告审阅、复杂案例归因时使用 Qwen3-Max。

8.5 百炼接法：OpenAI Chat Completions 兼容接口

百炼官方说明提供多种接口，其中 OpenAI Chat Completion 社区生态成熟、迁移和集成第三方工具成本最低；并明确支持 OpenAI 兼容接口，只需要调整 API Key、BASE_URL 和模型名即可迁移。对本项目，优先采用 OpenAI Chat Completions 兼容接口，而不是先上 Responses API。

8.6 必开能力：Function Calling + JSON Mode

百炼官方明确说明 Function Calling 的工作流是：模型返回 JSON 工具调用指令，应用执行工具，再把工具结果回填上下文；结构化输出（JSON Mode）通过设置 response_format={"type":"json_object"} 可以确保返回标准 JSON，减少下游解析失败。对本项目，这两项都必须开启。

8.7 暂不纳入 v0.1
DeerFlow 2.0
DeepAgents
RAG
浏览器自动化
分布式消息队列
大规模 memory 系统

理由：当前目标只有一个——稳定。

十六、主要风险与应对
风险 1：把 LLM 当成真相生成器

后果：不稳定、不可回放、不可审计。
应对：所有官方 payload 只能来自状态机、fixtures、validators、emitters。

风险 2：一开始追求 100% 拟真

后果：项目失控，迟迟无法联调。
应对：先做最小必要返回结构、常见状态、常见错误码，再逐步提高拟真度。这也是你说明文档原本的建议。

风险 3：把 provider 和 unified 混在一起

后果：未来无法平滑切 real provider。
应对：坚持 mock/real provider 分离，unified 层保持稳定。

风险 4：过早引入复杂 agent/runtime

后果：核心仿真不稳定。
应对：v0.1 不引入 DeerFlow 2.0 / DeepAgents 做核心运行时，只保留 LangGraph 编排。

风险 5：JD / 快手公开字段细节不足

后果：前期字段级拟真不完整。
应对：v0.1 先按公开能力域模拟，等后续拿到控制台文档或沙箱样本再补细节。京东和快手公开页已能确认能力域，但字段细节确实不如淘宝/抖店/企微透明。

十七、最终拍板建议

如果现在就要拍板，我建议：

项目名称：模拟官方 API 返回内容与推送行为的多平台仿真 Agent
项目定位：多平台官方行为仿真层 + 多平台客服中台
核心框架：FastAPI + Pydantic + PostgreSQL + Redis + LangGraph
默认模型：百炼 qwen3.5-plus

一、数据库设计的总体原则

这套数据库不是普通业务库，它要同时支持：

状态推进
官方工件生成
推送/回调回放
运行审计
结果复现
后续 real provider 切换验证

所以表设计必须分成两层：

第一层：状态真相层
保存“当前世界状态是什么”，例如订单、物流、售后、会话、消息、鉴权状态。

第二层：官方工件层
保存“平台对外吐出了什么”，例如 HTTP 查询响应、webhook、callback、sync_msg 缓冲、评估报告。

这套分法和你现有 OpenAPI 是对齐的：YAML 已经把 OrderCore、OrderItem、ShipmentNode、ShipmentResponse、AfterSaleResponse、WeComMessage、PlatformWebhookEnvelope 这些对象定义出来了，所以数据库不应该再发明第四套对象体系，而应该围绕这些统一对象建模。

从 PostgreSQL 能力上看，这样设计也合适：PostgreSQL 原生支持 uuid 类型和 gen_random_uuid() 这类 UUID 生成函数，支持 jsonb 存结构化 JSON 并保证值符合 JSON 规则，还支持 B-tree 和 GIN 等多种索引类型；其中 B-tree 是默认索引，GIN 适合索引复合值，例如 jsonb 这类结构。

二、数据库选型与存储策略
2.1 主库

PostgreSQL

原因不是“大家都用”，而是它正好适合这类系统：

关系明确：run、event、snapshot、artifact 有强关联
结构灵活：很多平台原始 payload 适合放 jsonb
可索引：对 status、platform、run_id、event_type 等查得很稳
可回放：事务和约束都比较适合做审计链路

jsonb 的意义很直接：你既能存固定列，也能存平台特定字段、原始包、headers、raw payload，同时 PostgreSQL 对 JSON 类型提供专门函数和操作符。

2.2 缓存 / 短期状态

Redis

这层不做最终真相，只做：

推送重试游标
短期 queue
幂等键
callback cursor
最近运行缓存
2.3 大工件

文件系统或对象存储

如果某些原始工件太大，例如大批量回放包、完整调试日志、测试报告附件，可以库里只存引用路径。

四、migration 顺序

这部分很重要。
不要一上来把所有表一起建，然后再想依赖关系。最稳的顺序是分 6 轮。

因为 PostgreSQL 当前文档已经把 uuid 和 gen_random_uuid() 作为原生能力暴露出来，所以这一层可以直接用。

Migration 005：工件与推送

再建：

五、字段设计总原则

为了避免后期乱掉，我建议统一遵守下面四条：

5.1 业务 ID 和库内主键分开

例如：

id：库内主键，UUID
order_id / after_sale_id / message_id：平台业务 ID

不要把平台业务 ID 直接当主键。

5.2 时间分三类
created_at / updated_at：库内维护时间
*_at_platform：平台业务发生时间
sent_at / acked_at：工件投递时间
5.3 平台原始包永远有落点

每个可能有平台差异的对象都留一个 raw_json / extra_json。

5.4 统一对象优先，平台特有字段后置

固定列只放稳定的统一语义，平台差异放 jsonb。这和你 OpenAPI 的统一模型设计是一致的。

七、这版 SQL 为什么这样设计
7.1 为什么大量使用 jsonb

因为你的系统同时有：

统一字段
平台特有字段
原始 payload
headers
callback body
调试信息

这些很难全用纯关系列表达。PostgreSQL 文档明确说明 JSON 类型可以保证值是有效 JSON，而 jsonb 还配合一系列 JSON 函数、操作符和索引能力使用。

7.2 为什么仍然保留大量固定列

因为你要做的是“中台真相”和“官方工件”的可比对、可查询，而不是把所有东西都塞进一大坨 JSON。
例如：

platform
order_id
status
event_type
artifact_type
route_key

这些都应该是固定列，便于稳定查询和索引。

7.3 为什么要单独有 artifacts

因为状态不等于工件。
同一个状态，可以产出：

查询接口响应
webhook
callback
sync_msg 种子
评估报告

把工件单列出来，你才能做真正的“官方行为回放”。

7.4 为什么要单独有 state_snapshots

如果你只存 event，不存 snapshot，后期调试会非常痛。
snapshot 能让你直接看到“第 12 步以后订单状态变成了什么”，而不用从头重放 11 个事件。

7.5 为什么 push_events 单独建表

因为推送是你项目里的一等公民。
你自己的 YAML 已经专门有 /mock/webhooks/{platform} 和 PlatformWebhookEnvelope，这说明推送回放不是附属功能，而是系统核心之一。

八、建议再加的约束

这几个约束我建议你在第二轮 migration 里补上。

8.1 平台约束

所有 platform 列统一约束为：

jd
taobao
douyin-shop
wecom-kf
xhs
kuaishou

因为这是你现有 OpenAPI 已经定义死的枚举范围。

8.2 状态约束

对几个关键状态列加 CHECK 或代码层 enum：

run.status
conversation.status
push_events.status

业务状态机更复杂的平台状态，建议先放代码层 enum，不急着全部塞进数据库 enum。

8.3 金额约束

金额列统一：

NUMERIC(18,2)
CHECK (amount >= 0)
8.4 逻辑完整性

例如：

sent_at 不能早于 created_at
acked_at 不能早于 sent_at
retry_count >= 0

app/domain/run_service.py
from __future__ import annotations

from datetime import datetime, timezone
from typing import Any
from uuid import uuid4

from app.models.artifacts import HttpArtifact, PushArtifact, RunArtifacts
from app.models.run import RunStatus, SimulationRun, SimulationRunAdvance, SimulationRunCreate


class RunService:
    """
    这是一版最小内存实现。
    目的：先把 official-sim/runs 跑通。
    后面再替换成 PostgreSQL + Redis。
    """

    def __init__(self) -> None:
        self._runs: dict[str, SimulationRun] = {}
        self._artifacts: dict[str, RunArtifacts] = {}

    def create_run(self, payload: SimulationRunCreate) -> SimulationRun:
        run_id = f"run_{uuid4().hex[:12]}"
        now = datetime.now(timezone.utc)

        initial_state: dict[str, Any] = {
            "auth_state": {"token_valid": True},
            "order_state": {},
            "shipment_state": {},
            "after_sale_state": {},
            "conversation_state": {},
        }

        run = SimulationRun(
            run_id=run_id,
            platform=payload.platform,
            scenario_name=payload.scenario_name,
            status=RunStatus.CREATED,
            strict_mode=payload.strict_mode,
            push_enabled=payload.push_enabled,
            current_step=0,
            state=initial_state,
            metadata=payload.metadata,
            created_at=now,
            updated_at=now,
        )
        self._runs[run_id] = run

        self._artifacts[run_id] = RunArtifacts(
            run_id=run_id,
            http_snapshots=[
                HttpArtifact(
                    method="POST",
                    path="/official-sim/runs",
                    status_code=201,
                    body={
                        "code": 0,
                        "message": "run created",
                        "data": {
                            "runId": run_id,
                            "platform": payload.platform,
                            "scenarioName": payload.scenario_name,
                        },
                    },
                )
            ],
            debug_notes=["Run created with in-memory storage."],
        )
        return run

    def get_run(self, run_id: str) -> SimulationRun | None:
        return self._runs.get(run_id)

    def advance_run(self, run_id: str, payload: SimulationRunAdvance) -> SimulationRun | None:
        run = self._runs.get(run_id)
        if run is None:
            return None

        run.current_step += 1
        run.status = RunStatus.RUNNING
        run.updated_at = datetime.now(timezone.utc)

        # 这里只做最小状态推进，占位用
        run.state["last_action"] = {
            "type": payload.action_type,
            "payload": payload.payload,
            "step": run.current_step,
        }

        artifacts = self._artifacts[run_id]
        artifacts.http_snapshots.append(
            HttpArtifact(
                method="POST",
                path=f"/official-sim/runs/{run_id}/advance",
                status_code=200,
                body={
                    "code": 0,
                    "message": "run advanced",
                    "data": {
                        "runId": run_id,
                        "step": run.current_step,
                        "actionType": payload.action_type,
                    },
                },
            )
        )

        if run.push_enabled:
            artifacts.push_events.append(
                PushArtifact(
                    event_id=f"evt_{uuid4().hex[:10]}",
                    event_type="simulation.run.advanced",
                    platform=run.platform,
                    body={
                        "runId": run_id,
                        "step": run.current_step,
                        "actionType": payload.action_type,
                    },
                    headers={"x-simulated": "true"},
                )
            )

        return run

    def get_artifacts(self, run_id: str) -> RunArtifacts | None:
        return self._artifacts.get(run_id)


run_service = RunService()
app/api/routes/runs.py
from __future__ import annotations

from fastapi import APIRouter, HTTPException, status

from app.domain.run_service import run_service
from app.models.artifacts import RunArtifacts
from app.models.run import SimulationRun, SimulationRunAdvance, SimulationRunCreate

router = APIRouter(prefix="/official-sim/runs", tags=["OfficialSimRuns"])


@router.post("", response_model=SimulationRun, status_code=status.HTTP_201_CREATED)
async def create_run(payload: SimulationRunCreate) -> SimulationRun:
    """
    创建一次仿真运行。
    """
    return run_service.create_run(payload)


@router.get("/{run_id}", response_model=SimulationRun)
async def get_run(run_id: str) -> SimulationRun:
    """
    获取运行详情。
    """
    run = run_service.get_run(run_id)
    if run is None:
        raise HTTPException(status_code=404, detail=f"Run not found: {run_id}")
    return run


@router.post("/{run_id}/advance", response_model=SimulationRun)
async def advance_run(run_id: str, payload: SimulationRunAdvance) -> SimulationRun:
    """
    推进一步状态机。
    """
    run = run_service.advance_run(run_id, payload)
    if run is None:
        raise HTTPException(status_code=404, detail=f"Run not found: {run_id}")
    return run


@router.get("/{run_id}/artifacts", response_model=RunArtifacts)
async def get_run_artifacts(run_id: str) -> RunArtifacts:
    """
    获取当前运行已产生的官方工件。
    """
    artifacts = run_service.get_artifacts(run_id)
    if artifacts is None:
        raise HTTPException(status_code=404, detail=f"Artifacts not found for run: {run_id}")
    return artifacts
