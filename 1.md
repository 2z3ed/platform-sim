一、不要做“一个 agent 干所有事”

最好拆成这 3 个核心模块：

1. World / Scenario Engine

这是真相层。
它维护：

user
order
shipment
refund / after_sale
conversation
emotion
current_step
platform_state

它不靠 LLM 决定真相，而是靠状态机 + fixtures + rules。
这点很重要，因为如果你把订单状态、物流状态、退款状态也交给 LLM 自由编，它就不再是“官方行为仿真”了。你前面整个项目计划书里反复强调的也是这个边界。

2. User Simulator Agent

这是你图里那个 agent。
它做的不是定义世界状态，而是：

读当前用户可见事实
判断“真实用户现在最可能问什么”
生成自然语言消息
决定要不要追问、升级情绪、要求人工、提退款

这层就是标准的 tool-calling agent。OpenAI 官方也明确把这种模式定义为：模型通过工具访问系统外部数据和功能，再决定如何响应。

3. Platform Behavior Layer

这层负责把状态引擎里的事实，变成：

官方订单查询返回
官方物流返回
官方退款返回
webhook / callback / push payload

也就是你前面一直说的 official-sim / 官方行为仿真层。
这一层和 user agent 是解耦的：agent 负责“用户怎么说”，platform layer 负责“平台怎么回”。

二、你这类 agent 最好的职责边界

你现在这个设计里，get_order / get_refund / get_shipment / list_orders 这些工具方向没问题。

A. 给 LLM 暴露的“用户视角工具”

这样生成出来的话术会更像人。

三、我建议你把 agent 做成“两阶段输出”，不要一步到位直接出文本

你图里现在是：

LLM 决策
调工具
直接生成用户消息

这能跑，但不够稳。
更好的做法是两阶段：

阶段 1：先产出结构化“用户意图决策”

例如：

{
  "selected_user_id": "u_001",
  "selected_order_id": "JD_ORDER_001",
  "intent": "ask_refund_progress",
  "emotion": "impatient",
  "need_tool_calls": [
    {"name": "get_order_summary", "arguments": {"order_id": "JD_ORDER_001"}},
    {"name": "get_refund_summary", "arguments": {"order_id": "JD_ORDER_001"}}
  ],
  "reason": "订单已付款且退款中，用户最可能追问退款进度"
}
阶段 2：再根据工具结果生成用户话术

例如：

{
  "user_message": "我的退款怎么还没处理好？订单号 JD_ORDER_001，已经好几天了。",
  "intent": "ask_refund_progress",
  "emotion": "impatient",
  "referenced_order_id": "JD_ORDER_001"
}

这样设计有两个好处：

你能审计 agent 为什么这么问
你能单独测“决策层”和“话术层”

而且如果你用 Structured Outputs，比纯 JSON mode 更稳，因为官方文档明确说：JSON mode 只能保证“是有效 JSON”，但不能保证一定符合你定义的 schema；Structured Outputs 才能更可靠地约束到你的 schema。

1. 文档目标

本文档定义 user-simulator-agent 的职责、边界、模块结构、工具 schema、状态 schema、编排流程、API 草图与落地顺序。

该 agent 的目标不是模拟“客服怎么答”，而是模拟：

真实用户在不同平台、不同订单/物流/退款/会话状态下
会怎么问
会什么时候追问
会不会升级情绪
会不会要求转人工
会不会重复催单 / 催物流 / 催退款

同时，该 agent 不能定义平台真相。
平台真相仍由 official-sim 层维护；agent 只能读取真相并生成用户行为。这个边界与当前项目总体架构一致：official-sim 负责官方行为真相，provider 负责适配，unified 负责对上提供稳定接口，AI 只负责编排、动作生成和报告。

2. 一句话定位

User Simulator Agent = 用户行为生成层。

它在整个系统中的位置是：

上游读取 official-sim 和 unified 已有的事实
中间用 LLM + tools 决定用户最可能说什么
下游把“用户消息 / 用户动作”注入到中台或仿真 run 中

它不是：

官方 payload 生成器
平台状态机
订单/退款/物流真相定义器
客服回复 agent
3. 设计原则
3.1 真相与话术分离

必须严格区分：

真相层：订单状态、物流状态、退款状态、会话状态、平台推送状态
话术层：用户如何表达、情绪如何升级、是否引用订单号、是否要求人工

真相层由 official-sim 决定；话术层由 user agent 决定。
LLM 只负责后者，不负责前者。

3.2 先决策，后发话

不要让模型一步直接输出最终消息。
推荐两阶段：

结构化决策：选哪个用户、哪张订单、什么意图、什么情绪、需要哪些工具
自然语言生成：基于工具结果生成真实用户消息

因为 Structured Outputs 比普通 JSON mode 更适合保证 schema 合规。

3.3 用户只能看见“用户视角事实”

对 LLM 暴露的工具，必须是“用户视角”的工具，而不是平台内部原始字段 dump。
例如优先提供 get_order_summary()，而不是直接把底层平台所有 raw fields 喂给模型。

3.4 编排优先用 LangGraph

这个 agent 是长时、有状态、可回放的流程，适合用 LangGraph 来做节点化编排与状态管理。LangGraph 官方定位就是 low-level orchestration runtime，适合 long-running、stateful agents。

4. 在整体系统中的位置

整体链路应为：

用户行为生成
→ official-sim 推进平台状态
→ 生成官方查询返回 / callback / webhook / push
→ provider 消费并标准化
→ unified 输出统一对象
→ AI / 前端 / 测试台消费统一对象

其中 user-simulator-agent 只做最前面的“用户行为生成与注入”。
现有 /mock/unified/orders/...、/mock/unified/shipments/...、/mock/unified/after-sales/...、/mock/unified/conversations/{conversationId}/suggest-reply 这些统一层接口应继续保留，作为上层消费面。

5. 核心模块

建议拆成 5 个模块。

5.1 Scenario Context Loader

负责加载当前 run 的上下文：

user profile
platform
conversation_id
selected order / refund / shipment
current step
recent messages
current emotional state
candidate intents
5.2 Tool Registry

负责向 LLM 暴露可调用工具的 schema。
这些 schema 采用标准 function calling 风格。OpenAI 的 function calling 正是让模型决定是否调用工具、由应用执行工具、再把结果回传模型。

5.3 Decision Engine

负责第一阶段的结构化决策：

选哪个用户
选哪个订单
当前最可能的用户问题是什么
情绪是什么
需要查哪些工具

输出必须是结构化 schema。

5.4 Message Renderer

负责把决策与工具结果，渲染成自然语言用户消息。
例如：

“怎么还没发货？”
“我的退款怎么还没到账？”
“系统显示签收了，但我根本没收到。”
5.5 Message Emitter

负责把生成的用户消息真正写入系统：

写入 run event
写入 conversation message
触发下一步 official-sim 状态推进
记录 artifact / audit log

  "user_message": "我的退款怎么还没处理好？订单号 JD_ORDER_001，已经等了好几天了。",
  "reason": "用户已知订单号，并且退款处理中，追问退款进度符合当前状态"
}
9. Prompt 设计

建议系统 prompt 写成这样：

你是一个“平台用户行为模拟器”。

你的任务不是扮演客服，也不是生成官方平台字段。
你的任务是：
1. 基于用户资料、订单/物流/退款/会话等工具结果，
2. 判断真实用户在当前状态下最可能会问什么，
3. 生成一条口语化、自然、短小的用户消息。

约束：
- 你不能编造订单状态、物流状态、退款状态。
- 你只能依据工具返回结果说话。
- 你只能模拟用户，不要替客服解释。
- 优先选择用户最关心、最自然的问题。
- 先输出结构化决策，再输出最终用户消息。
- 消息长度 1 到 3 句话。
- 允许引用订单号，但不要长篇解释。
- 如果当前状态不适合发问，可以输出 no_action。

你可以使用可用工具来获取用户可见事实。
10. LangGraph 节点设计

建议 v1 graph 节点如下：

load_context
select_user
select_order_or_case
decide_intent
call_tools
render_message
validate_message
emit_message
record_artifact
end

LangGraph 适合这种设计，因为它专门面向 long-running、stateful agent orchestration。

10.1 节点说明
load_context

读取 run state、候选用户、最近消息、平台类型、当前场景。

select_user

若 run 已绑定用户则直接使用；否则从候选用户中选一个。

select_order_or_case

决定当前围绕哪张订单、哪张退款单、哪段物流、哪段会话展开。

decide_intent

输出结构化意图。例如：

ask_order_status
ask_shipment
ask_refund_progress
complain_not_received
escalate_to_human
product_question
call_tools

调用相应 summary 工具获取事实。

render_message

将 intent + tool results 转成自然语言消息。

validate_message

检查：

是否与工具结果矛盾
是否越权说了客服视角
是否引用了不存在的 ID
是否过长
是否缺少情绪/意图标签
emit_message

真正写入当前会话。

record_artifact

记录本次 agent 决策与消息产物，用于回放和报告。

13. 与 official-sim 的衔接方式

推荐这样接：

user-simulator-agent 不直接改平台状态
它只发“用户动作 / 用户消息”
这些消息进入 official-sim run
official-sim 再根据状态机决定是否：
触发订单查询返回变化
触发退款状态变化
触发 callback / webhook / push
让 unified 层拿到新的上下文

这样就能保持边界稳定：

user-simulator-agent = 用户行为生成
official-sim = 平台真相与行为仿真
provider/unified = 对上统一消费。

15. 非目标

当前阶段不做：

让 LLM 直接生成官方 API payload
让 user agent 控制平台状态机
复杂多 agent 互相协商
复杂 RAG
浏览器自动化
DeepAgents / DeerFlow 作为 P0 核心运行时

这也和你前面整个项目的非目标保持一致：先把稳定仿真层打稳，再让外层 agent 使用它。

16. 成功标准

这个 agent 在 P0 算成功，需要满足：

能从当前 run 上下文中选用户和订单
能通过 tools 获取用户可见事实
能生成与事实一致的用户消息
能把消息写入系统
能被 official-sim 和 unified 层消费
能被回放、审计、测试
17. 最终拍板建议

User Simulator Agent 不是系统主框架，而是 official-sim 外层的“用户行为生成器”。

最终技术定位建议是：

主系统：FastAPI + 状态机 + fixtures + emitter + validator
Agent 编排：LangGraph
Tool/schema/prompt 封装：LangChain 薄适配层
LLM：只负责用户话术、意图决策、报告总结
官方 payload 真相：严格由 official-sim 定义。
